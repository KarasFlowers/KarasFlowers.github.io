NginX的异步非阻塞机制
哈桑农户
  2026-01-05 12:00:00   2026-05-09 17:01:17  
  • 编程迷思
  

Nginx 之所以能用很少的资源处理大量并发连接，核心在于它的 异步非阻塞 架构。但”异步非阻塞”到底是什么意思？和传统的”同步阻塞”有什么区别？

同步阻塞：一个服务员只服务一桌

传统服务器（如 Apache 的 prefork 模型）的工作方式：

  1. 来了一个客人（请求）
  2. 服务员（线程）全程陪同：点菜 → 等厨房做 → 上菜 → 结账
  3. 在等厨房做菜的时候，服务员就站在那桌旁边干等
  4. 这桌没吃完，服务员不能去服务下一桌

如果有100桌客人同时来，就需要100个服务员（线程）。每个服务员都占内存，100个服务员就消耗大量资源。

异步非阻塞：一个服务员服务所有桌

Nginx 的工作方式：

  1. 来了一个客人，服务员记下”这桌要红烧肉”
  2. 不等厨房做完，立刻去服务下一桌
  3. 厨房做好了喊一声”红烧肉好了！”
  4. 服务员听到后再去上菜

整个过程中，服务员（进程）从来不”等”。他一直在忙——要么接新客人，要么给做好菜的桌上菜。

类比：你去餐厅吃饭，好服务员不会站在厨房门口等你的菜。他会去给其他桌点菜、倒水、结账。你的菜好了，厨房会叫他。

关键概念

同步 vs 异步

  • 同步：发出请求后，一直等到结果返回才继续做下一件事
  • 异步：发出请求后，不等结果，先去做别的事，结果好了再处理

阻塞 vs 非阻塞

  • 阻塞：在等待期间，线程被”卡住”了，什么都做不了
  • 非阻塞：在等待期间，线程可以去做其他事情

Nginx 的组合是 异步 + 非阻塞：不等结果 + 等待期间能做其他事。

Nginx 的具体实现

Nginx 使用 事件驱动（Event-Driven） 模型：

底层依赖操作系统的 epoll（Linux）/ kqueue（macOS）机制，可以高效地监控成千上万个连接的状态变化。

为什么 Nginx 比 Apache 快？

Apache (prefork) Nginx
模型 一个连接一个进程 一个进程处理所有连接
1000个并发 需要1000个进程 1个进程就够了
内存消耗 高 低
等待时 进程干等 处理其他连接

这就是为什么 Nginx 在高并发场景下性能远超传统服务器。
